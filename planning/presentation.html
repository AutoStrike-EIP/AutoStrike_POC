Architecture Hexagonale

autostrike/ ├── server/ # Backend Go │ ├── cmd/autostrike/ # main.go │ ├── internal/ │ │ ├── domain/ # Logique metier │ │ ├── application/ # Use cases │ │ └── infrastructure/ # HTTP, DB, WS │ └── configs/ # Techniques YAML │ ├── agent/ # Agent Rust │ └── src/ │ ├── main.rs │ └── executor.rs # Execution cmds │ ├── dashboard/ # Frontend React │ └── src/ │ ├── pages/ # 10+ pages │ ├── components/ # UI components │ └── lib/ # api.ts │ └── docs/ # MkDocs

Clean Architecture

Les dependances vont toujours vers l'interieur. Le domain ne connait pas la DB ni HTTP.

  • domain/ = logique pure, pas de deps externes
  • application/ = orchestration des use cases
  • infrastructure/ = adapters (HTTP, DB, WS)

L'avantage

Changer SQLite → PostgreSQL = 1 fichier.
Changer le framework HTTP = 1 dossier.
Le code metier ne bouge pas.

Pourquoi pas juste utiliser Caldera ?

Caldera c'est Python, c'est plus lourd a deployer, et il n'a pas de scoring, pas de scheduling, pas d'analytics. Notre agent Rust est plus leger et plus discret qu'un agent Python. Et on a une architecture hexagonale propre qui permet d'evoluer facilement.

Questions Probables

"Pourquoi pas Python pour l'agent ?"

"Python c'est facile a detecter et bloquer par les antivirus. Rust compile en binaire natif, c'est plus discret et performant."

"C'est quoi le Safe Mode ?"

"L'agent n'execute pas les commandes, il simule juste. C'est pour tester sans risquer de casser quelque chose."

"Comment on ajoute une technique ?"

"Fichier YAML dans configs/techniques/ avec l'ID MITRE, le nom, la tactique, et les commandes par OS."

"C'est securise ?"

"JWT + RBAC pour l'API, mTLS pour les agents en prod, AGENT_SECRET pour l'auth agents, rate limiting, CSP, HSTS."

"Comment les tests ?"

"Vitest pour le front avec mocks. Tests Go classiques pour le back avec mocks des repositories. 780+ tests au total."

"Pourquoi un seul port ?"

"Simplifie le deploiement et la config firewall. Le serveur Go sert tout : fichiers statiques, API REST, WebSockets."
